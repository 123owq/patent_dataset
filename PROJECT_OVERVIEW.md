

## 전체 구현 흐름

```
[데이터 준비]
      │
      ├─ 1. KIPRIS에서 원천 데이터 수집 
      │       특허 PDF, 의견제출통지서 PDF, 인용특허 PDF, 보정서 JSON
      │
      ├─ 2. PDF 파싱 → 구조화된 JSON <-- **현재단계**
      │       정보손실 0%를 목표로 모든 텍스트·표·구조 보존
      │
      └─ 3. 툴별 학습/평가 데이터셋 구축
              Tool1(거절이유 분석), Tool2(청구항 파싱),
              Tool3(클레임 차트), Tool4(차이점, 전략), Tool5(보정안) 각각의
              입출력 쌍으로 데이터셋 구성

[성능 개선]
      │
      ├─ Phase 1: DSPy 프레임워크 프롬프트 최적화
      │       데이터셋으로 프롬프트 자동 최적화 → 성능 평가
      │       (일부 평가 항목은 변리사 직접 검토)
      │       → 이것만으로 충분하다면 여기서 완료
      │
      └─ Phase 2: LoRA 파인튜닝 (필요시)
              DSPy 프롬프트 최적화의 한계가 확인될 경우
              ~1,000건 데이터셋으로 llm 모델 파인튜닝
              → 파인튜닝 후 성능 평가 지표로 검증

[성능평가지표] 
      │
      │─ tool1, 2, 3 은 llm 및 기존 데이터셋으로 평가 가능
      │
      │─ tool4, 5 (차이점 전략, 보정항) 에 대한 평가는 변리사만 가능한 상황
```

## 툴 완성도 판단 기준

이 프로젝트의 모든 툴(Tool1~5)은 구현 방향 자체가 **LLM 성능에 전적으로 의존**. 코드가 하는 일은 LLM에 프롬프트를 넘기고 응답을 파싱하는 것이 전부이기 때문에
코드 레벨에서 툴의 완성도를 판단하는 것은 불가능

따라서 툴 완성도는 다음 두 단계의 **성능평가지표**로만 판단 가능함:

| 단계 | 방법 | 판단 기준 |
|---|---|---|
| 1차 | 프롬프팅 엔지니어링 후 성능 평가 | 목표 지표 달성 시 → 툴 완성으로 판단, 종료 |
| 2차 | 모델 학습(LoRA) 후 성능 평가 | 1차에서 한계가 확인된 툴에 한해 진행, 지표로 재판단 |

## 데이터 수집 → 파싱 → 데이터셋 구축 상세 흐름

```
KIPRIS Plus API
    │
    ├── 특허 공보 PDF          → patent_extract.py   → patent_parsed.json
    ├── 의견제출통지서 PDF     → extract_v6.py       → oa_parsed.json
    ├── 인용특허 PDF (한/일)   → jp_extract.py       → jp_parsed.json
    └── 보정서                 → (이미 구조화 JSON)  → 그대로 활용
    │
    ▼
각 툴 입출력 쌍 구성
    │
    ├── Tool1 데이터셋: OA 텍스트 + 인용 법조항 실제 조문 → {거절유형, 법조항, 인용특허, 심사관논리}
    ├── Tool2 데이터셋: 청구항 텍스트 → {구성요소, 독립/종속 구분, 인용관계}
    ├── Tool3 데이터셋: 당사 구성요소 + 선행특허 전문 → {O/X 판정 + 근거}  ← 레이블 자동 생성 가능
    ├── Tool4 데이터셋: 클레임차트 + 거절이유 → {대응전략, 반박포인트}
    └── Tool5 데이터셋: 원본청구항 + 전략 → {보정안, 명세서 근거}
```

## 수집 원천 데이터

KIPRIS Plus API로부터 5종의 데이터를 수집합니다.

| 수집 항목 | 형식 | 용도 |
|---|---|---|
| 특허 공보 | PDF | 청구항·발명설명 추출 → Tool2/3/5 학습 입력 |
| 의견제출통지서 | PDF | 거절이유 추출 → Tool1/3/4 학습 입력 |
| 특허법 조문 | 텍스트 (별도 준비) | OA에서 심사관이 인용한 법조항(§29②, §42 등)의 실제 조문 — OA 기반 데이터셋의 컨텍스트로 필요 |
| 인용특허 (한국/일본) | PDF | 선행특허 구성요소 추출 → Tool3 학습 입력 |
| 보정서 | JSON (API 직접 제공) | 보정 패턴 학습 → Tool5 학습 정답 |
| 청구항 변동이력 | JSON | 출원~등록 전 과정 추적 |

- 이어받기 지원: `kipris_done.txt`로 완료 추적
- 출력 폴더: `kipris_api/output/{출원번호}_{발명명칭}/`
